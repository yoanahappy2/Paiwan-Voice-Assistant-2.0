## 1. Prompt Engineering（提示工程）

**來源：** aneasystone.com 提示工程筆記（極詳盡的中文綜述）

### 核心概念
提示詞由四個要素組成：
- **指令（Instruction）**：給模型下達任務
- **上下文（Context）**：額外背景信息
- **輸入數據（Input Data）**：用戶輸入
- **輸出指示（Output Indicator）**：指定輸出格式

### 基本原則
1. 從簡單開始，迭代優化（零樣本→少樣本→微調）
2. 指令放在開頭，用分隔符（### 或 """）隔開
3. 描述要明確、具體、詳細
4. 通過示例明確輸出格式
5. 避免說「不要做什麼」，改說「要做什麼」
6. 角色扮演（Role Prompting）可以顯著改善輸出

### 提示詞框架
- **CRISPE 框架**：Capacity/Role, Insight, Statement, Personality, Experiment
- **LangGPT 結構化提示詞**：Role → Profile → Skill → Rules → Workflow

### 關鍵技術
| 技術 | 說明 |
|------|------|
| 零樣本提示 | 直接輸入，不加示例 |
| 少樣本提示 | 提供幾個示例引導模型（利用上下文學習） |
| 指令提示 | 直接告訴模型意圖（需指令微調過的模型） |
| 思維鏈（CoT） | 展示推理步驟，提升推理能力（需≥100B模型） |
| 零樣本 CoT | 加「Let's think step by step」即可 |
| 自我一致性 | 多次 CoT 投票選最一致答案（溫度0.5，採樣5次+） |
| 最少到最多（LtM） | 分治法：先拆子問題，再逐步解決 |
| 思維樹（ToT） | 樹狀搜索多條推理路徑，支持回溯 |

### 對我們專案的啟發
- **vuvu Maliq 的 System Prompt 就是用 Role Prompting**
- 我們的 CoT（Chain-of-Thought）展示已經在做
- RAG 本質上是在提供「上下文（Context）」

## 2. Fine-tuning（微調）/ LoRA

**核心：** 在已有模型上用特定數據再訓練，讓模型更適合某任務
- LoRA 只訓練少量低秩參數，大幅降低成本
- **我們已經在用：** Whisper-tiny + LoRA 微調排灣語 ASR

### 與 RAG 的取捨
- 微調改變模型參數（模型變聰明），但成本高
- RAG 不改變模型（模型沒變聰明），但成本低
- **我們的策略：先用 RAG（成本低、夠用），遠期再做 CPT/SFT**

---

## 3. RAG（檢索增強生成）

**核心流程：** 從向量數據庫返回相似文本 → 結合輸入給模型
- **缺點：** 沒有更改底層模型，模型本身沒有變聰明
- **操作注意：** 切片策略、多模態數據、檢索方式
- **我們的現狀：** FAISS + 智譜 embedding-3，117 句語料，Top-3 準確率 90%

---

## 4. Function Calling（函數調用）

讓大模型按約定格式輸出調用指令，由外部系統執行
- **模型從「只會說話」變為「會調用工具」**
- 飛書 Bot 的消息收發本質上就是 Function Calling
- OpenAI、智譜等主流平台都支持

---

## 5. MCP（Model Context Protocol）

**核心：** 標準化協議，讓 AI 以統一方式連接外部工具/數據源/服務
- 像USB-C一樣標準化了 AI 與資源的連接
- **最重要的貢獻：** 工具可以跨 AI 應用復用
- 架構：Host（AI應用）→ Client（MCP 客戶端）→ Server（各種資源）
- 安全性：細粒度權限、沙盒執行

### 對我們專案的意義
- 未來飛書 Bot 的工具整合可以考慮 MCP 標準
- 但短期（11天）不需要考慮 MCP，直接用 API 就好

## 6. Agent（智能體）

**來源：** tw93.fun「你不知道的 Agent：原理、架構與工程實踐」（⭐ 極高質量）

### Agent Loop 核心邏輯
```
while (模型返回 tool_use) {
    執行工具 → 把結果喂回模型
}
返回最終文本
```
- 感知→決策→行動→反饋，不斷循環
- 新能力通過三種方式接入：擴展工具集、調整系統提示、狀態外化

### Workflow vs Agent
| 維度 | Workflow | Agent |
|------|----------|-------|
| 控制權 | 代碼預定義 | LLM 動態決策 |
| 執行方式 | 工具順序固定 | 工具按需選擇 |
| 適用 | 流程固定、邊界清晰 | 需要推理與靈活判斷 |

**我們的飛書 Bot 介於兩者之間** — 有固定流程（接收→處理→回覆），但 LLM 決策部分是 Agent 行為

### 五種常見控制模式
1. **提示鏈（Prompt Chaining）**：順序步驟
2. **路由（Routing）**：輸入分類，走不同分支
3. **並行（Parallelization）**：子任務並發/投票
4. **編排器-工作者（Orchestrator-Workers）**：中央 LLM 分解任務
5. **評估器-優化器（Evaluator-Optimizer）**：生成→評估→循環

### ⭐ 為什麼 Harness 比模型更關鍵
Harness = 圍繞 Agent 構建的測試、驗證與約束基礎設施
- 驗收基線、執行邊界、反饋信號、回退手段
- **OpenAI 案例：3 個工程師 5 個月百萬行代碼**，靠的不是模型強，而是工程決策對了
- 「Agent 看不到的內容等於不存在」
- 「約束編碼化而非文檔化」
- 「確定性邏輯不進上下文，交給外部系統」

### 上下文工程（Context Engineering）
- **Context Rot**：無關內容佔上下文大頭時，決策質量下滑
- 信息按頻率和穩定性分層：
  - **常駐層**：身份定義、禁止項（短、硬、可執行）
  - **按需加載**：Skills 和領域知識
  - **運行時注入**：動態信息
  - **記憶層**：跨會話經驗（MEMORY.md）
  - **系統層**：Hooks/代碼規則處理確定性邏輯

### 三種壓縮策略
| 策略 | 成本 | 丟什麼 | 適用 |
|------|------|--------|------|
| 滑動窗口 | 極低 | 早期上下文 | 簡短對話 |
| LLM 摘要 | 中 | 細節，保留決策 | 長任務 |
| 工具結果替換 | 極低 | 工具原始輸出 | 工具密集型 |

### 工具設計原則
- 工具設計比數量更關鍵（5 個 MCP 服務器就可能帶來 55,000 tokens）
- **ACI（Agent-Computer Interface）原則**：
  - 工具對應 Agent 的目標，不是 API 的操作
  - 返回與下一步決策相關的字段
  - 錯誤要結構化，含修正建議
  - 描述要說明何時用、何時不用

### 記憶系統
- **上下文窗口**（工作記憶）
- **Skills**（程序性記憶）：怎麼做
- **JSONL 會話歷史**（情景記憶）：發生了什麼
- **MEMORY.md**（語義記憶）：重要事實

## 7. Multi-Agent（多智能體）

**來源：** Anthropic 官方博客

### 核心結論
> **先從單 Agent 開始，多 Agent 不是默認選項**

### 三個場景 Multi-Agent 優於單 Agent
1. **上下文保護**：子任務產生大量無關上下文時
2. **並行化**：需要同時探索多個方向時
3. **專業化**：不同任務需要不同工具集或系統提示時

### 多 Agent 的代價
- Token 消耗是單 Agent 的 **3-10 倍**
- 每個 Agent 是新的故障點
- 上下文在交接時容易丟失

### 對我們專案的啟發
- **飛書 Bot 用單 Agent 就夠了**，不需要多 Agent 架構
- ASR/TTS/RAG/LLM 各自獨立優化，已經是模組化設計

## 8. Context Engineering（上下文工程）

### 核心思想
> 提供給 LLM 的所有信息（對話歷史、用戶輸入、背景知識、工具結果）都是上下文
> 上下文工程關注如何高質量篩選、壓縮、組織上下文

### 最佳實踐
1. 常駊系統提示要短且穩定（利於 Prompt Caching）
2. 動態信息放在後面
3. 確定性邏輯不進上下文
4. 文件系統適合作為上下文接口（按需讀取）

## 9. Agent Skill（智能體技能）

**來源：** Anthropic 官方博客 + agentskills.io

### 核心概念
- Skill = 一個目錄 + SKILL.md 文件
- 包含指令、腳本、資源
- **漸進式披露（Progressive Disclosure）**：
  1. 系統提示只載入 name + description
  2. 需要時讀取完整 SKILL.md
  3. 更深層的引用文件按需載入

### 設計原則
- 描述要短但精準（說何時用/何時不用，加反例）
- 反例不是可選項，是關鍵（有反例準確率 53%→85%）
- 一次只載入一個 Skill
- 有副作用的 Skill 要顯式限制調用時機

### 生態
Agent Skills 已被廣泛採用：Claude Code、Cursor、GitHub Copilot、VS Code、Gemini CLI、OpenAI Codex 等 18+ 平台支持

### 對我們專案的啟發
- **飛書 Bot 可以設計成多個 Skill：**
  - learn（教學）、translate（翻譯）、quiz（測驗）、daily（每日一句）
  - 每個 Skill 有獨立的 SKILL.md
  - 系統提示只載入描述，需要時才讀取完整指令

## 11. Harness Engineering（駕馭工程）

**來源：** OpenAI 官方博客「工程技術：在智能體優先的世界中利用 Codex」

### 核心故事
3 個工程師 5 個月，百萬行代碼，1500 個 PR。**每行代碼都是 AI 寫的。**

### 關鍵經驗

**1. 給 Agent 一張地圖，不是一本 1000 頁的說明書**
- AGENTS.md 只保留約 100 行作為索引
- 細節拆到 docs/ 目錄按需引用
- 「一個巨大的指令文件會擠掉真正重要的信息」

**2. 約束編碼化，而非文檔化**
- 寫在文檔裡的規範容易被忽略
- 編碼進 Linter、類型系統、CI 規則才能強制執行
- 「當一切都重要時，一切都不重要了」

**3. Agent 端到端自主完成**
- 驗證→復現→修復→驗證→開 PR→回覆 Review→合併
- 全鏈路不需要人介入

**4. 知識必須存在於代碼庫**
- Agent 運行時無法訪問的內容等於不存在
- 「存儲在 Google Docs、聊天記錄或人們頭腦中的知識都無法被系統訪問」

**5. 垃圾回收式技術債清理**
- 定期運行後台 Agent 掃描偏差、更新質量等級
- 「技術債就像高息貸款，不斷小額償還比一次痛苦清償好」

### 對我們專案的啟發
- SPRINT_PLAN.md 就是我們的「地圖」
- Benchmark 就是「約束編碼化」的體現
- 每日檢查清單就是 Harness 的一部分

## 總結：對復賽的具體啟發

### 立即可用的概念

1. **Agent 定位**：飛書 Bot 是一個 Workflow + Agent 混合體
   - 固定流程（接收→處理→回覆）= Workflow
   - LLM 決策部分 = Agent

2. **上下文工程**：vuvu Maliq 的 System Prompt 要精簡
   - 常駐：角色定義 + 核心規則（短而硬）
   - 按需：教學內容、語法規則（觸發時才注入）
   - 動態：用戶輸入、RAG 檢索結果

3. **Skill 架構**：飛書 Bot 的指令設計
   - /learn, /translate, /quiz, /daily 各自獨立
   - 系統提示只載入描述，觸發時才讀取完整指令

4. **Harness 思維**：SPRINT_PLAN.md + Benchmark + 每日檢查
   - 每一步都有驗收標準
   - 約束量化（BLEU、WER、成功率 > 95%）

5. **不急著用 Multi-Agent**：單 Agent + 模組化足夠
